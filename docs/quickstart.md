# Quickstart

Go from a local install to a passing typed factory using nothing but this page — a real
`schema.json`, generated types, a `factory.ts` that consumes them, and a test that runs
green.

## 1. Install

Everything below happens inside a NEW folder for your factory package — create one first:

```sh
mkdir my-factories && cd my-factories
```

This folder is "your own package" for the rest of this guide: `schema.json`,
`tsconfig.json`, `factory.ts`, and `factory.test.ts` all end up here later, next to
each other.

### `bun link` (recommended)

Inside the SDK's own checkout:

```sh
bun run link:sdk
```

`link:sdk` is `bun run build && bun link` collapsed into one documented command — it builds
`dist/` fresh, then registers the package in Bun's global link store. Then, inside your new
consumer folder:

```sh
bun link @pbuilder/sdk
```

No separate `bun init` is needed first — `bun link` creates a minimal `package.json` and
`node_modules` for you if the folder doesn't already have one.

Re-run `bun run link:sdk` after any `src/` change — the link points at `dist/`, not `src/`,
so it does not pick up edits automatically.

### Tarball (alternative)

If you would rather not use `bun link`, pack a tarball and install it directly (still inside
your consumer folder):

```sh
bun pm pack
bun add ./pbuilder-sdk-0.0.0.tgz
```

Neither path installs from a live registry — `@pbuilder/sdk` is not published there yet.

## 2. Configure your consumer

Add TypeScript and Bun's ambient types as dev dependencies:

```sh
bun add -d typescript @types/bun
```

Then add a `tsconfig.json` next to `schema.json`:

```json filename=tsconfig.json
{
  "compilerOptions": {
    "lib": ["ESNext"],
    "target": "ESNext",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "allowImportingTsExtensions": true,
    "strict": true,
    "noEmit": true,
    "skipLibCheck": true,
    "types": ["bun"]
  },
  "include": ["*.ts"]
}
```

Three settings here are load-bearing, not boilerplate:

- `moduleResolution: "NodeNext"` (or `"bundler"`) — TypeScript's default legacy resolution
  cannot read a package's `exports` map, which is the ONLY route to `@pbuilder/sdk`'s five
  subpaths (`./testing`, `./commons`, and friends). Without this, your editor reports
  `Cannot find module '@pbuilder/sdk/commons'` even though the import is correct.
- `allowImportingTsExtensions: true` — step 5 imports the generated types with an explicit
  `.ts` extension (`./schema.generated.ts`), which TypeScript rejects by default.
- `types: ["bun"]` — step 6's `factory.test.ts` imports from `"bun:test"`. Leaving `types`
  empty or unset hides that ambient module even with `@types/bun` installed; naming it
  explicitly is what makes `Cannot find module 'bun:test'` go away.

Skip this step and step 5's/step 6's `import` lines will show as editor/type errors, even
though `bun test` (which strips types rather than checking them) still runs.

## 3. Define your schema

Every factory reads its typed inputs from an adjacent `schema.json`:

```json filename=schema.json
{
  "properties": {
    "serviceName": { "type": "string", "label": "Service name", "required": true },
    "port": { "type": "number", "label": "Server port", "required": true }
  }
}
```

## 4. Generate types

Run the codegen bin against your package directory (the one positional argument it takes —
no flags):

```sh
pbuilder-codegen .
```

This writes `schema.generated.ts` next to `schema.json`, exporting `type Input`:

```ts
// AUTO-GENERATED by pbuilder-codegen — do not edit. Regenerate: pbuilder-codegen <package-dir>
// @schema-digest sha256:…

export type Input = {
  /** Service name */
  serviceName: string;
  /** Server port */
  port: number;
};
```

## 5. Write your factory

Create `factory.ts` next to `schema.json`:

```ts filename=factory.ts
import { create } from "@pbuilder/sdk/commons";
import type { Input } from "./schema.generated.ts";

export const run = (input: Input) => {
  create(`src/config/${input.serviceName}.ts`, {
    template: `export const port = ${input.port};`,
    options: {},
  });
};
```

A factory is a plain function typed against the GENERATED `Input` — never a hand-written
generic. `create` is imported from `@pbuilder/sdk/commons`, the same entry every
author-facing directive comes from. The next step runs `run` through `runFactoryForTest`,
which drives it against an in-memory fake — the factory itself never imports anything from
`./testing`.

## 6. Test it

Create `factory.test.ts` next to `factory.ts`:

```ts filename=factory.test.ts
import { test, expect } from "bun:test";
import { runFactoryForTest } from "@pbuilder/sdk/testing";
import { run } from "./factory.ts";

test("factory creates the service config file", async () => {
  const result = await runFactoryForTest(run, { serviceName: "api", port: 3000 });

  expect(result.error).toBeUndefined();
  expect(result.tree.get("src/config/api.ts")).toEqual("export const port = 3000;");
});
```

```sh
bun test factory.test.ts
```

If this passes, you have a working, typed factory built entirely from generated types —
no hand-written input shape anywhere.
